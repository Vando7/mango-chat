# Agent Guidelines — chat-app

## Rules
- **Always keep this file up to date.** Every time you modify code, configs, or architecture, update the relevant section below. Outdated docs cause mistakes.
- Follow the file structure described below. New files go in `src/` under `api/` or `components/`.
- Use JSX consistently. No new `.ts` files unless explicitly requested.
- Run `npx vite build` to verify before confirming changes work.
- The Vite dev server proxies `/v1/**` to the LLM backend — don't add CORS handling in the frontend.
- **Never push to git on your own accord.** Always ask the user first before running `git push`. You may commit changes without asking, but pushing requires explicit approval.

## Architecture

- **`src/App.jsx`** — Main app component, manages all state and orchestration. Lucide icons (`PanelLeft`/`PanelLeftClose`/`Settings`/`X`/`Sparkles`), gradient brand mark, glassy backdrop-blur header. The model list is fully sourced from the connected backend — no hardcoded fallback entries. Initial state is `models: []`, `selectedModel: ''`; the send button is gated on a non-empty `selectedModel`.
- **`src/api/client.js`** — API client. `fetchModels` tries LM Studio's `/api/v0/models` first (rich metadata: `type`, `publisher`, `arch`, `quantization`, `state`, `maxContext`, `loadedContext`, `capabilities`); falls back to OpenAI `/v1/models` returning `{id}` objects only. `chat` async generator yields `{text, reasoning, toolCalls, finishReason}` per chunk and accepts a 4th `tools` arg (OpenAI function-tool array) — when present, `tool_choice: 'auto'` is sent. Streamed `delta.tool_calls[]` are accumulated index-keyed (each chunk concatenates `function.arguments` partial JSON onto the previous). When the backend doesn't surface structured `tool_calls[]` and the model emits inline XML markup instead (Nemotron-style: `<think>…</think>`, `<tool_call><function=name><parameter=k>v</parameter></function></tool_call>`), the exported `extractNemotronMarkup` parser pulls the markup out of `delta.content`, lifts `<think>` content into the reasoning channel, converts `<tool_call>` blocks into the same accumulator shape as native tool calls, and overrides `finishReason` to `'tool_calls'` so the multi-turn loop kicks in. Native `delta.tool_calls[]` always wins when present (no regression for well-behaved backends). Partial tags at chunk boundaries are held back rather than leaked to the visible bubble. Also exports `buildToolCatalogPrompt(tools)` — renders the OpenAI function-tool array as a human-readable system-prompt block describing each tool's name, one-line description, and parameter schema, plus instructions to emit calls in the inline-XML dialect that `extractNemotronMarkup` understands. Used by `App.jsx` when "force tools" is on. `setApiBase` mutates the shared base URL. Always returns *objects*, never bare strings — callers must read `m.id` for the model name to send.
- **`src/components/MessageList.jsx`** — Message rendering, auto-scroll (forwardRef), animated reasoning verb (mexican wave + dot cycle), hidden after streaming ends. Empty state uses Lucide `MessagesSquare` with mango glow; assistant avatar uses gradient `Sparkles`. Assistant messages show the model id as a small gray label above the bubble. The bubble itself renders all `images[]` as a `flex-wrap` row of thumbnails, then any `toolCalls[]` as collapsible `ToolCallBlock` rows (Lucide `Wrench` icon, status pill: `Receiving args`/`Running`/`Done`/`Error`; expand reveals pretty-printed JSON args + result), then the markdown content. Per-message action row (`MessageActions` subcomponent) appears under each bubble on hover: `< n/N >` version arrows when `totalVersions > 1`, `Regenerate` button on the latest assistant message only (hidden while streaming), and a `Trash2` delete button that removes the message *and everything after it*. Uses `key={position-version}` so React reconciles cleanly when the active version of a slot changes.
- **`src/components/ChatInput.jsx`** — Pill-style composer (focus-glow + ring), Lucide icons (`Paperclip`/`ArrowUp`/`Square`/`X`), gradient send button, attach hover-rotate animation. Multi-image support: file input has `multiple`, attachments render as a thumbnail strip above the composer (each with its own remove button), and the textarea has an `onPaste` handler so `Ctrl+V` of a clipboard image (or multiple images) appends them to the queue. `imageUrls: string[]` is the canonical prop; `onPasteImages`/`onRemoveImageAt(i)` are the callbacks.
- **`src/components/Sidebar.jsx`** — Left sidebar with chat history, search-with-icon, `MessageSquarePlus` new-chat, `Trash2` delete, `ChevronLeft` collapse handle. All Lucide icons.
- **`src/components/SettingsPanel.jsx`** — Floating settings *window* anchored top-right (absolute, z-30) of the chat pane. Header with minimize/close, glassy backdrop-blur, mango-gradient header tint. Body has four stacked sections: **Server** row (with `Server` icon + reconnect button), **System Prompt** preset list (radio-style rows including a "None" entry, `+ New` button, hover-revealed edit/delete per row, inline `PresetEditor` for create/edit with name input + content textarea), **MCP Tools** indicator + inline editor (flat tool count + per-server `McpServerRow` with status pill `ready`/`error`, tool count, expand-to-see-tool-list; header has a `sent to model`/`model has no tool_use`/`none` pill plus an `Edit` button that opens the inline `McpConfigEditor` — monospace textarea preloaded from `GET /mcp/config`, validates JSON on save, posts to `POST /mcp/config` which respawns servers in-process; on success the parent App applies the returned `{ tools, servers }` directly via `onMcpConfigSaved`), and **Model** card grid. Each `ModelCard` shows type badge (LLM/VLM/EMB), loaded-state pulse dot, quantization, publisher · arch, max-context (formatted K/M), `tools` capability chip. Embeddings are rendered as disabled. Includes a filter input over the grid. Selected card has a check badge + mango ring.
- **`src/api/db.js`** — SQLite database layer using sql.js (browser-based SQLite), IndexedDB persistence. Schema versioned via `PRAGMA user_version` with const `SCHEMA_VERSION` (currently **6**) — when bumped, all tables are dropped and recreated on next init (no migration; chat history is disposable). Tables: `chats`, `messages`, and `presets`. Messages are stored as `(position, version, is_active)` rows so each conversation slot can have multiple regenerated drafts. Each row also stores the `model` id that produced it (rendered above assistant bubbles), an `images` column holding a JSON-encoded array of data URLs (`encodeImages`/`decodeImages`), and a `tool_calls` column holding a JSON-encoded array of executed tool-call records (`encodeToolCalls`/`decodeToolCalls`) — each shaped `{id, name, args, result, status, error?}` with status in `done`/`error` for persisted rows (`streaming-args`/`executing` are runtime-only and demoted to `error` on rehydrate). Presets store user-defined system prompts (`id, name, content, created_at, updated_at`); the *active* preset id is **not** stored here — it lives in `localStorage` under key `mango.activePresetId` (session UI state, not chat data). Helpers: `appendMessage`/`addNewVersion` take optional trailing `toolCalls`; `updateMessageVersion(chatId, position, version, content, reasoning, toolCalls?)` — passing `toolCalls` undefined leaves the column untouched. `setActiveVersion` (flip arrows), `deleteFromPosition` (remove slot + tail), `getMessageVersions` (returns `{versions[pos][ver], active[pos]}`), `listPresets`/`createPreset`/`updatePreset`/`deletePreset`.
- **`vite.config.js`** — Vite dev server with `/v1` proxy to LM Studio (`http://172.27.112.1:1234`); also loads `mcpPlugin()` from `vite-plugin-mcp.js` which exposes `/mcp/tools` and `/mcp/call`.
- **`vite-plugin-mcp.js`** — MCP bridge as a Vite middleware plugin. On dev-server start it reads `mcp.json` from the project root, spawns each declared server as a stdio child, runs the JSON-RPC handshake (`initialize` → `notifications/initialized` → `tools/list`), and exposes four endpoints:
  - `GET /mcp/tools` → `{ tools: [...flattened OpenAI function-tool shape], servers: [{name, ready, tools, error}] }`. Tools are renamed to `mcp__<server>__<tool>` so the LLM gets a flat namespace.
  - `POST /mcp/call` → body `{ name, arguments }` → strips the `mcp__` prefix, routes to the matching server via `tools/call`, returns `{ content: '<joined text>', isError: bool, raw }`. MCP `content` arrays are flattened by joining each `text` part.
  - `GET /mcp/config` → `{ exists, content, path }` returning the on-disk text of `mcp.json` plus the absolute path the bridge resolved to.
  - `POST /mcp/config` body `{ content: '<json text>' }` → validates JSON, rejects non-object top-level / non-object `mcpServers`, atomically writes the file (`mcp.json.tmp` + rename) reformatted with 2-space indent, kills all current children, respawns from the new config, awaits the handshake, and returns `{ ok, tools, servers }` so the frontend can apply the new state without a second round-trip.
  Internal state (handles + initPromise) is captured in a closure object so middleware sees the latest set of children after any hot-reload. Per-request 15s timeout. Children are killed on `httpServer.close`, `exit`, `SIGINT`/`SIGTERM`/`SIGHUP`. Missing/empty `mcp.json` is a clean no-op (endpoints return empty list).
- **`mcp.json`** — Claude-Desktop-compatible MCP server config. Top-level `mcpServers: { <name>: { command, args, env? } }`. Paths in `args` are relative to the project root (resolved by the plugin via `server.config.root`).
- **`mcp-servers/`** — Local MCP server implementations + a project-local Python venv (`mcp-servers/.venv/`, gitignored).
  - `date_server.py` — FastMCP server exposing `now()` (ISO 8601 + tz + epoch) and `today()`.
  - `sysinfo_server.py` — FastMCP server exposing `get_info()` (platform, system, release, machine, processor, python_version, cpu_count, hostname, server_uptime_seconds).
  - `requirements.txt` — `mcp>=1.0`. Install once via `python3 -m venv mcp-servers/.venv && mcp-servers/.venv/bin/python3 -m pip install -r mcp-servers/requirements.txt`. The `mcp.json` invokes the venv's interpreter explicitly.
- **`src/api/mcp.js`** — Browser client for the bridge: `fetchMcpTools()` returns `{ tools, servers }` (or empty arrays on failure — degrades silently when the bridge is offline), `callMcpTool(name, args, signal)` returns `{ content, isError }`, `fetchMcpConfig()` returns `{ exists, content, path }`, and `saveMcpConfig(content)` POSTs new text and resolves with `{ ok, tools, servers }` describing the post-restart state. The save helper rejects on validation/HTTP errors so the editor can surface a useful message.
- **`tailwind.config.js`** — Tailwind CSS config; defines `mango.{50..900,bg,panel}` color tokens and animations (`cursor-blink`, `fade-up`, `fade-in`).
- **`public/favicon.svg`** — Mango-gradient rounded-square chat-bubble mark.
- **`dev.sh`** — Dev runner script (installs deps if missing, runs `npm run dev`)
- **`scripts/`** — CLI test scripts: `check.sh`, `test-api.sh`, `test-stream.js`, `test-image.js`

## Default backend & model picker

- `App.jsx` defaults `serverUrl` to `http://172.27.112.1:1234` (LM Studio) and auto-connects on mount. The default URL is just an initial value — when running on a different machine, the user types the new URL into the Server input and clicks Reload.
- The model list is whatever the connected backend reports. There is **no hardcoded fallback model** — if the backend has zero models the picker is empty and the send button stays disabled (gated on a non-empty `selectedModel`).
- **Auto-follow loaded model**: the selected model mirrors whatever LM Studio currently has loaded. While `connected`, App.jsx polls `fetchModels()` every 5s (paused on `document.hidden`, resumed on visibility), updates the cards' state, and auto-switches `selectedModel` to whichever chat (`type !== 'embeddings'`) model is in `state: 'loaded'`. Switching models in LM Studio while the app is open updates the selection within ~5s without user action. If no chat model is loaded but the previously-selected id has disappeared from the list, the polling effect falls back to `pickDefaultModel(list)`.
- Helpers `pickDefaultModel`, `pickLoadedChatModel`, `sameModelList` in `App.jsx` operate on rich model *objects* (`{id, type, state, ...}`). Keep `pickLoadedChatModel` and the polling effect in sync if changing what counts as "the loaded model."

## MCP (Tool Calls)

- **Layout**: `mcp.json` (repo root) → `vite-plugin-mcp.js` (dev-server middleware bridge) → `src/api/mcp.js` (browser client) → `App.jsx` (multi-turn loop) → `MessageList.jsx` (`ToolCallBlock` rendering). Python servers live in `mcp-servers/` with their own venv at `mcp-servers/.venv/` (gitignored).
- **Tool naming**: every tool is renamed to `mcp__<server>__<tool>` so the LLM sees a flat namespace and the bridge can route a `tools/call` back from the name alone. The `McpServerRow` UI strips the `mcp__<server>__` prefix when displaying the local tool name.
- **Enabled when**: `mcpTools.length > 0` AND *either* the active model's `capabilities` array contains `tool_use` (helper `modelSupportsTools` in `App.jsx`) *or* the user has flipped the **force tools** toggle on the model card (helper `toolsActiveFor`, set persisted in `localStorage` under `mango.forcedToolModels`). When forced, `runStream` also prepends `buildToolCatalogPrompt(mcpTools)` to the first system message of the request — the inline-XML parser in `client.js` then lifts any `<tool_call>` markup out of `delta.content` so backends that drop the OpenAI `tools` request field (e.g. Lemonade) can still drive MCP. No per-server toggle yet — every configured server is sent. `SettingsPanel`'s MCP section is read-only and shows a header pill (`sent to model` / `model has no tool_use` / `none`); the per-model `force tools` chip on each `ModelCard` (Lucide `Zap`) reflects the override.
- **Multi-turn loop** (`runStream` in `App.jsx`): on each iteration calls `chat(workingHistory, model, signal, mcpTools)`, accumulates streamed text/reasoning/tool_calls. If the closing `finish_reason === 'tool_calls'`, each call is parsed (JSON.parse args, default `{}`), executed serially via `callMcpTool`, recorded into `executedToolCalls` with `status: 'streaming-args' → 'executing' → 'done'/'error'`, and pushed back into `workingHistory` as an assistant turn (with `tool_calls`) followed by `{role:'tool', tool_call_id, content}` rows. Loops up to `MAX_TOOL_TURNS = 8`. Cumulative text from all turns is joined with `\n\n` separators into the assistant version's `content`; cumulative tool calls go into `toolCalls` (persisted as JSON via `db.js`).
- **Streaming**: while args are still arriving, the corresponding `ToolCallBlock` shows `status: 'streaming-args'` with the partial JSON args (often invalid until the chunk completes — that's expected, we don't try to parse mid-stream).
- **Aborts**: the existing `AbortController` is also passed to `callMcpTool` so Stop kills both the LLM stream and any in-flight MCP call.
- **Bridge endpoints** (auto-served by Vite dev server):
  - `GET /mcp/tools` → `{ tools: [OpenAI function-tool, ...], servers: [{name, ready, tools, error}, ...] }`. Tool list is empty if no `mcp.json` or all servers failed init.
  - `POST /mcp/call` body `{ name: 'mcp__<srv>__<tool>', arguments }` → `{ content: '<joined text>', isError, raw }`. Bad name → 400. Unknown server → 404. Server not ready → 503.
- **Adding a new MCP server**: drop it in `mcp-servers/` (any language — must speak MCP stdio), then either edit `mcp.json` directly or use the in-app **Edit** button in the MCP Tools section of `SettingsPanel`. Saving via the UI hot-reloads servers in-process (no dev-server restart needed); editing the file directly still requires a `npm run dev` bounce.

## State Management

- All state lives in `App.jsx` — server URL, connection status, model list (rich objects), `selectedModel` (string id), `convo`, input, image URL, loading/error/streaming flags, `showSettings` (default `true`), `settingsMinimized`, `sidebarOpen`, `chatRefreshKey`, `presets` (list from DB), `activePresetId` (mirrored in `localStorage` via `setActivePresetId` helper), `mcpTools` (flat OpenAI function-tool array from `/mcp/tools`), `mcpServers` (per-server status array from the same endpoint), `forcedToolModels` (`Set<string>` of model ids the user opted in to prompt-injected tool calling, mirrored to `localStorage` under `mango.forcedToolModels` via `toggleForcedTools`).
- `convo = { versions, active }`: `versions[position]` is an array of drafts at that slot, each shaped `{ role, content, images: string[], reasoning?, model?, streaming? }`; `active[position]` is the version index currently displayed. The flat list rendered to `MessageList` is derived via `useMemo` as `versions.map((vs, p) => vs[active[p]])` plus `position`/`version`/`totalVersions` metadata. Each assistant version captures `selectedModel` at the moment it was generated so the displayed model name reflects what actually ran (useful when you regen the same prompt across different models).
- Streaming flag toggled on before the generator starts, off after completion or error. While streaming, `runStream` mutates `convo.versions[position][version].content` per chunk; persistence (`updateMessageVersion`) only fires once in the `finally` block.
- Error messages appear as red text inside the floating `SettingsPanel`.
- The `connected` state gates the auto-follow polling effect — set `true` after a successful `fetchModels`, reset on failure.

## Common Patterns

- **Persistence**: per-action — `appendMessage` on send (one row per user msg, one empty row per assistant slot), `addNewVersion` on regenerate, `updateMessageVersion` in `runStream`'s `finally` (covers normal completion, errors, aborts), `setActiveVersion` when the user clicks the version arrows, `deleteFromPosition` on the trash button. No global "rewrite all rows" call — each operation touches only the rows it needs.
- **Regenerate**: only the latest assistant slot is regenerable. `handleRegenerate(position)` calls `addNewVersion` to insert an empty draft, sets it active, builds the model history from the active path *up to but not including* `position`, then runs `runStream` into the new version index.
- **Version arrows / delete**: `handleSwitchVersion(position, idx)` updates `convo.active[position]` and persists via `setActiveVersion` — works on any slot, including frozen ones (it just changes what's displayed for that slot, not the rest of the conversation). `handleDeleteMessage(position)` truncates `convo.versions`/`active` to length `position` and calls `deleteFromPosition`. Both are no-ops while `streaming`.
- **System prompt prepending**: When `activePresetId` resolves to a preset with content, both `handleSend` and `handleRegenerate` push `{role:'system', content: preset.content, images: []}` at index 0 of `historyToSend` *before* sending. The preset is **not** persisted into the messages table — it's a session-global overlay applied to every request. `client.js`'s `chat()` mapper is shape-agnostic and passes the system row through verbatim.
- **Streaming**: `for await (const chunk of chat(history, model))` — accumulates text and reasoning separately
- **Image upload**: FileReader reads each file as a data URL (`data:image/...;base64,...`); images are stored as `string[]` per message, then `client.js` spreads each into a separate `{type: 'image_url', image_url: {url}}` content part for the OpenAI vision format. Paste-from-clipboard uses the same pipeline (`addImageFiles` is shared between file input + paste handler).
- **Models endpoint flexibility**: `fetchModels()` tries LM Studio's `/api/v0/models` first for rich metadata, falls back to plain OpenAI `/v1/models`. Both `{models:[]}` and `{data:[]}` shapes are accepted. Always returns objects (`{id, ...}`), never bare strings.
- **Reasoning content**: collected from `delta.reasoning_content`, accumulated separately from the main text. Shown live during streaming with the animated mexican-wave verb; collapses into a custom show/hide block (no `<details>`) once streaming ends.
- **Auto-follow polling**: `App.jsx` runs a 5s interval while `connected`, paused on `document.hidden`, with an `inFlight` guard. It diffs the model list via `sameModelList(prev, next)` to avoid spurious re-renders, and switches `selectedModel` to whichever chat model is in `state: 'loaded'`.

## Recent Changes

- **Imperative tool-call prompt + lift calls out of `<think>`** (2026-05-08): Two changes to make Qwen3 actually emit calls instead of just narrating intent. (1) `buildToolCatalogPrompt` rewritten in imperative voice with explicit do/don't few-shot examples ("your ENTIRE next response must be a single <tool_call> block — no preamble, no acknowledgment, no code fences"). (2) `extractNemotronMarkup` now scans the body of every `<think>…</think>` block for `<tool_call>…</tool_call>` markup via a new internal `liftCallsFromInner` helper, treats those as real calls (so they fire), and strips them from the reasoning channel — Qwen3 in thinking mode often emits its tool calls inside its `<think>` block, and previously they vanished into the reasoning column. Two new fixtures cover the in-think case.
- **Tool-call parser now accepts Qwen/Hermes JSON dialect** (2026-05-08): `extractNemotronMarkup` previously only recognized Nemotron-style nesting (`<function=name><parameter=k>v</parameter></function>`) inside `<tool_call>`; Qwen3 (and other Hermes-trained models) emit a JSON object instead — `<tool_call>{"name": "fn", "arguments": {...}}</tool_call>`. The internal `parseNemotronToolBlock` is renamed `parseToolCallBlock` and now tries `JSON.parse` first when the trimmed block starts with `{` (also accepts `parameters` as an alias for `arguments`), and falls back to the XML form. Synthetic ids changed from `nemotron_call_<idx>` → `xml_call_<idx>`. `buildToolCatalogPrompt` now documents both dialects so a model trained on either will produce something the parser catches. Without this fix, Qwen tool calls were surfaced to the user as raw `<tool_call>…</tool_call>` markup and the multi-turn loop never triggered (parser found no calls → `finishReason` stayed `'stop'`). 4 new fixtures in `scripts/test-tool-parser.mjs` cover the JSON shape (single-line, multi-line, `parameters` alias, malformed-json fallback).
- **Force prompt-injected tools per model** (2026-05-08): Lemonade and other OpenAI-compatible backends that don't pass the `tools` request field through to the model can now still drive MCP. Each `ModelCard` in `SettingsPanel` gains a `Zap`-icon "force tools" chip (right-aligned in the meta row); clicking it toggles a per-model override stored in `App.jsx` as a `Set<string>` (`forcedToolModels`), persisted to `localStorage` under `mango.forcedToolModels`. `App.jsx` exposes `toolsActiveFor(modelId, models, forcedToolModels)` (replaces the bare `modelSupportsTools` check at `runStream` time and the `mcpEnabledForModel` SettingsPanel pill). When forced is on, `runStream` (1) sends the OpenAI `tools` field anyway — backends that ignore it (Lemonade) drop it harmlessly; backends that honor it use it natively — and (2) prepends `buildToolCatalogPrompt(mcpTools)` (new export in `client.js`) to the first existing system message (or unshifts a new system row), describing every tool's name, one-line description, and parameter schema, plus instructions to emit calls in the inline-XML dialect (`<tool_call><function=name><parameter=k>v</parameter></function></tool_call>`). The existing `extractNemotronMarkup` parser then lifts those calls out of `delta.content` and the multi-turn loop runs unchanged. The catalog is merged into the same system message as any active preset (rather than two separate `system` rows) because some chat templates only render the first system message.
- **In-app `mcp.json` editor with hot-reload** (2026-05-08): Settings panel's MCP Tools section gains an Edit button that opens an inline `McpConfigEditor` (monospace textarea). Saving POSTs the new content to a new `POST /mcp/config` endpoint, which validates the JSON, atomically writes the file (`mcp.json.tmp` + `rename`), kills the current MCP children, respawns from the new config, awaits the JSON-RPC handshake, and returns `{ ok, tools, servers }` so the frontend can apply the new state without an extra round-trip. Companion `GET /mcp/config` returns `{ exists, content, path }`. Bridge state was refactored into a closure-captured object so middleware closures see the new handles after each restart. Frontend helpers `fetchMcpConfig` / `saveMcpConfig` in `src/api/mcp.js`. App.jsx adds `refreshMcpTools` + `handleMcpConfigSaved` (prefers the inline result; falls back to a `/mcp/tools` refetch).
- **Inline-XML tool-call/reasoning fallback parser** (2026-05-08): Some models (notably nvidia/nemotron-3-nano) emit tool calls and reasoning as XML inside `delta.content` instead of the structured `delta.tool_calls[]` / `delta.reasoning_content` SSE fields, because LM Studio's llama.cpp doesn't have a parser configured for their dialect. Added exported `extractNemotronMarkup` in `client.js` (re-parses `fullText` from scratch on every chunk; handles `<think>…</think>` and `<tool_call><function=name><parameter=k>v</parameter></function></tool_call>`; holds partial tags at chunk boundaries; surfaces malformed blocks to the user).
